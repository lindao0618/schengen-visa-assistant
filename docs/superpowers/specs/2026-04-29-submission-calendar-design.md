# 递签日程看板设计

日期：2026-04-29

## 背景

团队现在会同时处理多个客户、多个签证案件。递签时间如果散落在微信窗口、备注或单个申请人详情页里，主管和专员很容易漏看当天或未来几天需要处理的客户。

本功能要把所有客户的递签时间集中成一个工作看板，用于每日排班、查漏、跟进和快速进入客户档案。

## 目标

- 以 `VisaCase.slotTime` 作为唯一权威的递签时间字段。
- 在系统里提供一个“递签日程”入口，集中查看所有有递签时间的案件。
- 同时提供列表式看板和月历视图，满足“今天要做什么”和“整月排期如何”的两种使用方式。
- 显示未填写递签时间的进行中案件，方便主管发现资料或预约信息缺口。
- 显示已完成递签的案件，方便复盘和确认客户是否已经进入后续流程。
- 点击日程项可以直接进入对应申请人详情页，并定位到签证 Case。

## 非目标

- 第一版不新建重复的递签日期字段，避免和现有 `slotTime` 数据口径冲突。
- 第一版不做复杂排班系统，例如人员工时、假期、自动派单。
- 第一版不接入外部日历同步，例如 Outlook、Google Calendar、企业微信日历。

## 用户角色

- 老板：查看全局递签排期、逾期和未填写情况。
- 主管：按负责人、签证类型、日期筛选，检查团队是否漏跟。
- 专员：查看自己负责的递签日程和当天任务。
- 客服：只读查看递签安排，方便回复客户。

权限沿用现有申请人/Case 权限：能看某个申请人，就能在日程里看到对应案件；只能看自己客户的角色，不展示其他人的日程。

## 数据口径

主要字段：

- 递签时间：`VisaCase.slotTime`
- 客户：`ApplicantProfile.name`
- 案件类型：`VisaCase.caseType` / `visaType`
- 地区/城市：`VisaCase.applyRegion`、法国案件的 `tlsCity`
- 当前状态：`VisaCase.mainStatus` / `subStatus`
- 负责人：`VisaCase.assignedTo`
- 出行时间参考：`VisaCase.travelDate`
- 最近更新：`VisaCase.updatedAt`

过滤范围：

- 默认只展示未关闭、未完成的案件。
- 已完成递签案件默认隐藏，但可以通过快捷筛选打开；已关闭案件仍默认隐藏。
- 没有 `slotTime` 的进行中案件进入“未填写递签时间”分组。

## 页面入口

在“申请人档案”模块增加一个主入口：

- 顶部导航或申请人列表页 header 增加 `递签日程` 按钮。
- 路由建议为 `/applicants/schedule`。
- 页面标题使用“递签日程”，副标题说明“按 slot 时间集中查看客户递签安排”。

## 页面结构

页面顶部是筛选区：

- 日期范围：未来 3 天、未来 7 天、未来 15 天、未来 30 天。
- 视图切换：看板列表 / 月历。
- 负责人筛选：全部、我负责、指定专员。
- 签证类型筛选：美签、法签/申根、其他。
- 状态筛选：进行中、待递签、已递签、已完成、异常。
- 快捷筛选：未填写递签时间、已完成递签。

页面主体提供两个视图。

## 看板列表视图

默认进入列表视图，因为它最适合日常工作排查。

分组顺序：

1. 未来 3 天
2. 未来 7 天
3. 未来 15 天
4. 未来 30 天
5. 未填写递签时间
6. 已完成递签

每条日程卡片显示：

- 客户姓名
- 递签时间，精确到日期和时间
- 签证类型、城市/地区
- 当前状态
- 负责人
- 出行时间参考
- 快捷操作：打开档案、复制客户关键信息

高风险样式：

- 未来 3 天：蓝色强调
- 未来 7 天：浅蓝提示
- 未来 15 天：中性提示
- 未来 30 天：弱提示
- 未填写递签时间：黄色提示
- 已完成递签：绿色低风险提示

## 月历视图

月历用于整体排期规划。

功能：

- 按月份展示每天有多少个递签案件。
- 每个日期格展示最多 3 条客户摘要，超出显示“+N”。
- 点击日期打开当天右侧抽屉或下方列表，显示当天全部递签客户。
- 支持上一月、下一月、回到今天。
- 日期格中的案件颜色按状态或紧急程度区分。

月历不直接替代列表视图。月历负责“看排期密度”，列表负责“逐个处理”。

## 详情页联动

从日程项点击“打开档案”后：

- 跳转到 `/applicants/[id]?tab=cases&caseId=[caseId]`。
- 如果当前详情页支持选中 Case，则自动切换到对应 Case。
- 如果当前版本暂时无法自动选中 Case，至少进入申请人详情页并打开签证 Case 标签。

在申请人详情页中，`slot 信息` 仍然是编辑递签时间的唯一入口。日程页第一版只读展示，不直接编辑，减少误操作风险。

## 接口设计

新增轻量 API：

`GET /api/applicants/schedule`

参数：

- `from`：开始日期，ISO 日期字符串。
- `to`：结束日期，ISO 日期字符串。
- `assigneeId`：负责人筛选，可选。
- `visaType`：签证类型筛选，可选。
- `status`：状态筛选，可选。
- `includeMissingSlot`：是否包含未填写递签时间案件。
- `includeSubmitted`：是否包含已完成递签案件。

返回：

- `items`：有递签时间的案件列表。
- `missingSlotItems`：未填写递签时间的进行中案件。
- `submittedItems`：已完成递签的案件。
- `summary`：今日数量、未来 3 天数量、未来 7 天数量、逾期数量、未填写数量、已完成递签数量。

接口必须沿用现有权限过滤逻辑，不能返回用户无权查看的申请人或案件。

## 性能设计

- API 只查询日程需要的轻量字段，不返回完整申请人详情、材料、历史日志。
- 默认查询范围为未来 30 天，避免一次拉取过多历史数据。
- 前端按日期分组在客户端完成，减少后端复杂度。
- 页面可以使用短时客户端缓存，但手动刷新必须强制重新请求。

## 错误处理

- API 失败：显示“递签日程加载失败”，保留刷新按钮。
- 没有日程：显示空状态，引导用户去申请人详情页填写 slot 时间。
- 未填写递签时间：作为独立分组展示，不当作错误。
- 日期格式异常：API 返回 400，并提示日期范围无效。

## 测试计划

- 单元测试：日期范围解析、权限过滤参数、日程分组、逾期判断。
- 源码结构测试：新页面必须轻量加载，不把申请人详情页重组件打进首屏。
- API 测试：不同角色只能看到有权限的案件。
- 构建验证：`npx tsc -p tsconfig.test.json`、`node --test .test-dist/tests/*.test.js`、`npm run lint`、`npm run build`。

## 第一版交付范围

第一版实现以下内容：

- `/applicants/schedule` 页面。
- 看板列表视图。
- 月历视图。
- 顶部筛选区，日期范围为未来 3 天、7 天、15 天、30 天。
- `GET /api/applicants/schedule` 轻量接口。
- 从日程项跳转到申请人详情页 Case 标签。
- 未填写递签时间分组。
- 已完成递签快捷筛选和分组。

暂不实现：

- 日程页直接编辑 slot 时间。
- 外部日历同步。
- 自动提醒消息推送。
- 复杂排班和工作量统计。
