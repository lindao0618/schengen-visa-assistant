# Visa Ops Agent 设计

日期：2026-05-03

## 背景

当前系统已经具备申请人档案、签证案件、文件归档、法签/美签自动化任务、递签日程、Reminder 日志和邮件发送基础能力。团队下一步需要的是一个内部运营智能体，把这些能力串起来，减少人工查找、建档、归档、判断优先级和启动自动化的时间。

本设计定义一个网站内的 `Visa Ops Agent`。它是内部运营助理，不是面向客户的客服机器人。它负责理解运营指令、调用受控业务工具、生成确认卡片、整理每日工作和月历提醒。Codex 不进入生产业务操作链路，只作为开发维护助手，用于修改代码、修复自动化、分析日志和补测试。

`Visa Ops Agent` 必须接入大模型。没有大模型时，系统只能做固定规则自动化，无法可靠理解自然语言指令、处理模糊文件名、生成材料建议或总结失败原因。第一版建议接入 DeepSeek V4，同时保留模型提供商适配层，避免业务逻辑被某一个模型绑定。

## 目标

- 根据 Excel 文件名和内容自动识别签证类型、申请人姓名、预计出行/递签时间，并生成建档确认。
- 同名申请人优先合并到已有档案，只新增或选择案件。
- 支持缺人、同名多人、已有未完成同类案件、文件覆盖、低置信度解析等待确认队列。
- 支持网页内 AI 对话框，允许查询、修改、归档文件、启动法签/美签自动化、检查材料、生成每日工作建议。
- 支持 TLS/AIS/DS-160/法签自动化结果回写申请人档案、签证案件和材料文件。
- 以递签/月历视图作为内部提醒核心入口，展示未来 3/7/15/30 天安排。
- 每日生成“今天我需要干什么”的运营简报，按紧急程度排序。
- 第一版提醒只面向内部：后台待办、站内提醒、月历高亮、每日工作简报。
- 保持所有写入、覆盖、批量和真实自动化动作有明确权限和确认流程。

## 非目标

- 第一版不自动给客户发邮件、微信或短信提醒。
- 第一版不让 Agent 直接操作数据库表或绕过现有业务 API。
- 第一版不把 Codex 接入生产业务数据，不让 Codex 直接修改申请人档案。
- 第一版不做复杂客服会话、客户侧聊天、外部日历同步或排班系统。
- 第一版不做全量多 Agent 框架；先用一个主 Agent 和多组受控工具。
- 第一版不处理 `slotTime` 冲突。业务上认为真实 slot 时间不会冲突，新的同步结果直接作为权威时间进入确认/写入流程。

## 用户角色

- 老板：查看全局排期、紧急待办、自动化失败、团队常规汇报。
- 主管：处理待确认队列、分配和检查团队工作、确认高风险动作。
- 专员：上传 Excel、归档文件、启动自动化、查看自己负责的每日任务。
- 客服或只读角色：查询申请人状态、递签时间、材料缺口和自动化进度。

权限沿用现有申请人和案件权限。用户只能通过 Agent 查询或操作自己有权访问的申请人、案件和文件。

## 总体架构

系统分为四层：

1. 前端入口层
   - 申请人详情页右侧 AI 对话框。
   - Excel 上传后的建档确认卡。
   - 递签日程月历和列表视图。
   - 每日工作简报面板。
   - 待确认队列页面。

2. Agent 编排层
   - 新增网站后端接口，例如 `POST /api/ops-agent/chat`。
   - 接收用户指令、当前页面上下文、附件引用和会话历史。
   - 调用模型理解意图，并选择受控工具。
   - 将高风险动作转换成确认卡片，而不是直接执行。
   - 通过模型适配层调用 DeepSeek V4 或其他兼容模型。

3. 受控工具层
   - 每个工具只做一个明确业务动作。
   - 工具内部复用现有 API、Prisma 查询、文件工作流和自动化任务入口。
   - 工具必须检查权限、校验参数、记录审计日志。

4. 业务数据和自动化层
   - `ApplicantProfile`、`ApplicantFile`、`VisaCase`、`UsVisaTask`、`FrenchVisaTask`、`ReminderLog`。
   - 法签自动化、TLS 同步、美签 DS-160/AIS 自动化、材料审核和文件生成服务。
   - 定时任务或后台 worker 生成内部提醒和每日简报。

## 交互与操作体验规范

`Visa Ops Agent` 的目标是加速运营，任何增加切换成本、打断心流或让专员重复输入上下文的设计都应避免。签证运营通常同时面对微信、系统、Excel、材料文件和自动化日志，Agent 必须像工作台的一部分，而不是另一个需要维护的窗口。

### 输入体验：零切换与快捷触发

#### 选中文本即调用

在网站内任何可选中文本上，右键菜单应提供 Agent 动作：

- `发给 Ops Agent 建档`
- `让 Agent 提取信息`
- `生成进度摘要`
- `生成催办话术草稿`

点击后，右侧 Agent 侧边栏自动打开，选中文本已预填为输入内容，Agent 直接输出提取结果或确认卡，免去复制、切换、粘贴和重新描述上下文。

第一版网页只能可靠捕获网站内的选中文本。外部微信窗口、桌面 Excel 或本地文件管理器里的内容，第一版通过粘贴、拖拽文件或上传入口进入 Agent；如果未来需要 OS 级选中文本右键调用，应单独设计浏览器扩展、桌面助手或系统剪贴板集成。

#### 文件拖拽直连

客户详情页、文件管理页和 Agent 对话框都应支持拖拽文件。用户把文件拖入 Agent 后，Agent 使用当前页面上下文和文件元数据生成归档确认卡。

示例：

`这是卢本伟的美签护照扫描件，要归档到当前案件吗？`

确认卡应提供：

- `确认归档`
- `改成其他槽位`
- `选择其他申请人`
- `取消`

如果文件名乱码，Agent 不能只靠文件名判断，应结合当前页面上下文、文件内容提取结果和已有申请人信息。

#### 快捷指令面板

对话框输入框左侧提供 `/` 或闪电图标，点击后弹出当前上下文的高频动作。菜单内容按页面动态变化。

申请人详情页常用动作：

- 查缺漏
- 改递签时间
- 生成催办话术
- 启动法签自动化
- 启动美签自动化
- 检查材料能否递签

月历页常用动作：

- 看今天要处理谁
- 看未来 3 天高风险
- 列未填写递签时间
- 汇总某负责人任务

导入队列常用动作：

- 处理缺人文件
- 批量匹配申请人
- 合并到已有档案
- 只归档不建案

快捷动作点击后直接生成对应指令或确认卡，减少键盘输入。

### 输出体验：可操作的结构化卡片

Agent 回复不能只有大段纯文本。凡是涉及申请人、案件、文件、任务、待确认项和材料缺口的回复，都应优先用结构化卡片展示。

#### 结果深度链接

Agent 输出的申请人姓名、案件编号、任务 ID、文件名必须是可点击链接。默认在新标签页打开详情，避免当前页面跳转导致对话上下文丢失。

示例：

`已找到 3 个同名的卢本伟`

结果应展示三行候选人，每个候选人包含姓名、手机号尾号、护照尾号、当前案件和最近更新时间。姓名链接在新标签页打开申请人详情页；用户确认后可以回到对话框选择目标。

#### 内嵌式微操作卡片

材料检查结果应直接带操作按钮。

示例：

`卢本伟缺护照首页和在职证明`

卡片底部提供：

- `上传护照`
- `上传在职证明`
- `生成催办话术`
- `加入今日待办`

点击上传按钮后直接调起文件选择框，选完后自动归档到正确槽位，并把结果追加到当前对话流。

#### 一键复制

所有生成的催办话术、案件进度摘要、每日工作简报、客户沟通草稿和失败原因说明，右上角必须有明显的 `复制` 按钮。

复制内容应适配微信粘贴格式：

- 不包含 Markdown 表格边框。
- 不包含内部工具 ID。
- 不包含客户不应看到的备注、失败日志和内部判断。
- 保留清晰分段和关键时间。

### 确认机制：行内与不打断

#### 行内确认流

所有预览、修改、自动化启动和归档确认卡都应内嵌在对话流中。禁止使用阻断式居中模态弹窗作为默认确认方式。

示例：

用户说：`启动法签建表`

Agent 返回行内确认卡：

`即将为卢本伟启动法签建表，预计耗时 3 分钟。`

操作：

- `确认执行`
- `取消`
- `查看输入文件`

用户可以继续滚动查看上下文，想好后再点确认。

#### 高风险置顶

涉及批量操作、文件覆盖、批量启动自动化、批量生成材料包等高风险确认卡，可以临时置顶在对话框顶部，直到用户处理。

置顶卡应显示：

- 风险等级。
- 涉及人数或文件数。
- 将要修改的字段或覆盖的文件。
- `确认执行` 和 `取消`。

置顶卡不能遮挡整个页面，也不能阻止用户继续查看详情。

### 过程反馈：透明化与状态同步

#### 步骤拆解展示

长耗时任务禁止只显示全局 Loading。Agent 应逐步展示正在做什么，缓解等待焦虑，并在出错时定位卡点。

示例流程：

1. 正在读取张三档案。
2. 正在比对 12 项法签材料。
3. 发现缺失：行程单未上传。
4. 正在生成已有材料 PDF 合并包。
5. 材料包已生成，等待确认归档。

每一步应有明确状态：等待、进行中、完成、失败、跳过。

#### 跨页面状态同步

Agent 确认修改字段后，当前打开的申请人详情页、列表页、月历页应无刷新局部更新。

示例：

- 修改 `slotTime` 后，详情页时间字段立即更新。
- 归档文件后，材料区立即显示新文件。
- 自动化任务创建后，任务列表立即出现新任务。
- 案件状态推进后，月历和每日简报风险标记同步变化。

实现可使用前端事件总线、SSE、WebSocket 或数据缓存失效机制。关键要求是用户不需要手动刷新页面才能看到 Agent 的执行结果。

#### 全局异步通知

Agent 启动长耗时任务后，即使用户离开对话框页面，任务完成或失败时也应在系统右下角弹出轻量 Toast。

示例：

- `卢本伟的法签建表已完成`
- `王五的 AIS 注册失败：验证码超时`
- `张三材料包已生成，等待确认归档`

Toast 应支持点击跳转到相关申请人、任务或对话 thread，默认新标签页打开详情。

### 容错体验：宽容与回退

#### 修改与重生成

Agent 的每条回复右上角提供：

- `重新生成`
- `修改指令`

点击 `修改指令` 后，系统把用户上一条指令填回输入框，专员只需改错别字或补一句话，不需要重新构思。

重新生成不能重复执行写入动作。对于已经执行过的写入或自动化，重新生成只重新解释结果或生成新的文案，不得再次调用工具。

#### 上下文撤销

支持自然语言发起回退确认。

示例：

- `撤回上一步`
- `不对，改回刚才的值`
- `改回昨天那个递签时间`

Agent 应结合审计日志识别可回退的字段和值，并生成回退确认卡。回退本身也必须经过确认并写入审计日志。

不能自动撤销的动作，例如真实网站提交、已发送外部消息、已覆盖且无备份的文件，应明确提示不可自动回退，并给出人工处理建议。

## Codex 与业务 Agent 边界

`Visa Ops Agent` 负责运行时业务操作：

- 查申请人。
- 预览 Excel 建档。
- 生成确认卡。
- 写入申请人和案件字段。
- 启动法签/美签任务。
- 查询任务进度。
- 生成内部待办和月历提醒。
- 检查材料完整度。

Codex 负责开发维护：

- 改网站代码。
- 新增或调整 Agent 工具。
- 修 Playwright/TLS/AIS 自动化脚本。
- 分析失败日志和截图。
- 补测试、修构建、改 UI。
- 生成设计文档、实施计划和 PR。

Codex 不应该作为生产后台常驻任务，也不应该被业务流程依赖。自动扫描、内部提醒、邮件发送和任务运行都应该由网站自己的后端、定时任务或 worker 完成。

## 大模型接入策略

网站内 Agent 的智能部分由大模型负责。它的职责是：

- 理解用户自然语言指令。
- 从页面上下文中识别当前申请人、案件、文件或日期。
- 判断应该调用哪个受控工具。
- 生成确认卡片、材料建议、失败摘要和每日工作简报。
- 在信息不足时提出澄清问题或创建待确认项。

模型不能直接写数据库，也不能直接访问文件系统。它只能输出结构化工具调用请求，由后端工具层验证权限、校验参数并执行。

### DeepSeek V4 作为第一版模型

第一版推荐使用 DeepSeek V4：

- 默认快速模型：`deepseek-v4-flash`
  - 用于日常对话、查询、建档解析、每日简报、待确认队列总结。
- 高质量模型：`deepseek-v4-pro`
  - 用于复杂材料判断、失败日志总结、批量导入歧义分析、长上下文案件复盘。

模型选择可以按任务自动切换：

- 普通查询和简短修改：Flash。
- 涉及长材料、复杂推理、自动化失败诊断：Pro。
- 成本或速率受限时：降级到 Flash，但保留确认流程。

### 模型提供商适配层

不要在业务代码里硬编码 DeepSeek。新增模型适配层，例如：

- `provider`: `deepseek`、`openai`、`qwen`、`anthropic`。
- `model`: `deepseek-v4-flash`、`deepseek-v4-pro` 等。
- `baseUrl`: 从环境变量读取。
- `apiKey`: 从环境变量读取。
- `supportsToolCalls`: 是否支持工具调用。
- `supportsJsonOutput`: 是否支持 JSON 输出。
- `supportsStreaming`: 是否支持流式输出。

建议环境变量：

- `OPS_AGENT_MODEL_PROVIDER=deepseek`
- `OPS_AGENT_MODEL=deepseek-v4-flash`
- `OPS_AGENT_REASONING_MODEL=deepseek-v4-pro`
- `DEEPSEEK_API_KEY=...`
- `DEEPSEEK_BASE_URL=https://api.deepseek.com`

密钥禁止写入源码。现有任何硬编码模型 key 都应该迁移到环境变量，并轮换已暴露的 key。

### Agent 调用模式

第一版使用后端编排，而不是前端直接调用模型：

1. 前端把用户指令、页面上下文和附件引用发送给 `/api/ops-agent/chat`。
2. 后端读取当前登录用户和权限。
3. 后端组装系统提示、业务上下文和可用工具列表。
4. 大模型返回工具调用或普通回复。
5. 后端执行受控工具。
6. 如需确认，后端返回确认卡；如无需确认，返回结果消息。

这样可以保证：

- API Key 不暴露到浏览器。
- 所有写入都经过后端权限检查。
- 工具调用有审计记录。
- 模型可以替换，前端不需要改。

### 异步与长耗时任务

启动自动化、生成材料包、读取大文件、同步 TLS/AIS 等长耗时工具不采用同步阻塞。工具层应立即返回 `taskId`、任务类型、初始状态和轮询/SSE 订阅信息。

前端通过现有任务轮询或 SSE 接收进度。任务完成后，Agent 可以把结果追加到当前对话流中，例如：

- 自动化任务成功，展示写回字段、生成文件和下一步建议。
- 自动化任务失败，展示失败原因、可重试动作和是否需要 Codex 在开发环境分析。
- 材料包生成完成，展示归档位置和缺失项。

长耗时任务的实际执行由后端 worker、已有任务系统或自动化服务完成，不能让模型请求长时间等待。

### 上下文与多模态约束

DeepSeek V4 支持长上下文，第一版可以放大上下文预算，但仍然必须按场景裁剪，不能把申请人的完整历史无差别塞进模型。

- 普通对话：注入当前申请人、当前激活案件、最近 10 条相关文件/任务/状态摘要。
- 案件复盘：注入当前申请人的全部案件摘要，以及当前案件完整关键历史。
- 材料检查：注入全部相关材料的结构化摘要，不直接塞原始图片或大文件。
- 失败诊断：注入最近失败日志摘要，并按任务配置附带有限数量的原始日志片段。
- 深度模式：用户明确要求完整分析某个申请人的全部历史时，才扩大到长上下文。
- 完整文件内容、长日志和 OCR 结果应先由后端压缩、脱敏并结构化后再进入 Prompt。

第一版 Agent 不直接处理图片流。涉及 `照片归档`、`护照扫描件归档` 等指令时，模型只接收文件 ID、文件名、文件槽位和业务校验结果。照片尺寸、人脸检测、背景检测、OCR、护照信息提取等由现有微服务完成，结果以文本和结构化字段形式提供给 Agent。

### 多对话支持

第一版应支持多对话，但每个对话必须有明确作用域，避免上下文串线。

建议的对话作用域：

- `applicant`: 绑定一个申请人，可选绑定当前案件。
- `case`: 绑定一个具体签证案件。
- `schedule`: 绑定递签日程页面和当前筛选条件。
- `import`: 绑定一次文件导入或待确认项。
- `global`: 后台首页或每日工作简报的全局运营对话。

每个对话保存：

- `conversationId`
- `scopeType`
- `applicantProfileId`
- `caseId`
- `confirmationItemId`
- `scheduleFilters`
- `createdByUserId`
- 最近消息摘要
- 最近工具调用轨迹

前端进入申请人详情页、月历页或导入流程时，应优先恢复对应作用域的最近对话；用户也可以新开对话。跨申请人、跨案件操作必须重新确认目标，不能沿用旧上下文直接写入。

## 文件命名和智能建档

### 法签 Excel 命名

格式：

`法签-姓名-预计出行时间-递签时间`

示例：

`法签-卢本伟-3月份出行-2.13-2.20（2.17去不了）`

解析结果：

- 签证类型：`france-schengen`
- 申请人姓名：`卢本伟`
- 预计出行时间：`3月份出行`
- 期望递签窗口：`2.13-2.20`
- 不可用日期：`2.17`
- 原始文件名：完整保留用于追溯

### 美签 Excel 命名

格式：

`美签-姓名-预计递签时间`

示例：

`美签-卢本伟-7月递签`

解析结果：

- 签证类型：`usa-visa`
- 申请人姓名：`卢本伟`
- 预计面签/递签月份：`7月`
- 原始文件名：完整保留用于追溯

### 年份推断

- 默认按当前年份解析。
- 如果解析出的日期已经早于当前日期，自动滚到下一年。
- 确认卡必须展示系统理解出的完整年份。

例如当前日期晚于 2026-02-20，文件名中的 `2.13-2.20` 会被解析为 2027-02-13 到 2027-02-20。

### 建档归属规则

- 同名申请人只有一个：默认合并到已有申请人档案。
- 同名申请人多个：进入待确认队列，让用户选择具体申请人。
- 没有姓名或无法识别姓名：进入“缺人文件”待确认，不能自动建档。
- 没有同名申请人：创建新申请人档案。
- 同一个申请人已有未完成同类型案件：必须弹出选择。
  - 更新当前未完成案件。
  - 新增一个新案件。
  - 只归档文件，不改案件。
  - 取消导入。
- 已有同类核心文件：不自动覆盖，必须确认替换或作为新案件材料保存。

## Excel 导入流程

1. 用户上传 Excel。
2. 系统解析文件名，提取签证类型、姓名、时间窗口和备注。
3. 系统读取 Excel 内容，提取护照号、手机号、邮箱、TLS/AIS/DS-160 等可校验信息。
4. 系统查找已有申请人和未完成案件。
5. 系统生成建档确认卡，展示：
   - 识别结果。
   - 匹配到的申请人。
   - 案件归属建议。
   - 文件归档槽位。
   - 需要人工确认的风险。
6. 用户确认后，系统创建或更新申请人、创建或选择案件、归档文件。
7. 系统记录审计日志和导入来源。

如果用户在对话框中修正信息，Agent 重新生成修改预览，用户确认后再写入。

## 对话框能力

### 页面上下文感知

对话框 API 必须接收前端传递的当前 URL、页面类型和资源 ID，例如 `currentApplicantId`、`currentCaseId`、`currentScheduleDate`、`currentConfirmationItemId`。

当用户指令存在歧义时，Agent 优先绑定当前页面上下文，减少表达成本。例如在卢本伟申请人详情页中，用户说 `把递签窗口改成 2.14-2.21`，系统应默认理解为当前申请人的当前激活案件，并生成确认卡；如果当前页面没有明确申请人或案件，则必须追问或进入待确认。

网页内 AI 对话框支持六类指令。

### 查询

示例：

- `卢本伟现在什么状态？`
- `他还缺什么材料？`
- `距离递签还有几天？`
- `最近自动化任务有没有失败？`

查询类动作可以直接返回结果，不需要确认。

### 批量名单处理

示例：

- `查一下这些人有没有建档：张三、李四、王五`
- 用户直接粘贴一列姓名或 Excel 复制出来的姓名列。
- `帮我看这些人谁还没有递签时间`

Agent 应批量查询并返回结构化状态汇总，而不是逐个返回对话气泡。默认输出 Markdown 表格，包含姓名、匹配结果、申请人 ID、当前案件状态、递签/面签时间、风险和建议动作。

批量名单处理只做查询、汇总和生成待办建议；涉及批量修改、批量启动自动化、批量生成材料包时必须进入确认流程。

### 修改信息

示例：

- `把卢本伟递签窗口改成 2.14-2.21`
- `2.18 去不了`
- `TLS 城市改成上海`
- `美签预计 7 月面签`

修改类动作先生成变更预览，用户确认后写入。

### 归档文件

示例：

- `把这个 Excel 归到卢本伟法签案件`
- `这张照片是他的美签照片`
- `不要覆盖旧文件，作为新案件材料`

涉及覆盖已有核心文件时必须确认。

### 启动自动化

示例：

- `给卢本伟启动法签建表`
- `生成回执 PDF`
- `提交法签最终表`
- `跑美签照片检测`
- `启动 DS-160 自动填表`
- `注册 AIS`

启动自动化必须确认，因为会消耗资源，可能影响真实账号状态。

### 材料指挥

示例：

- `检查卢本伟能不能递签`
- `给卢本伟生成递签材料包`
- `帮我列今天要催的人`
- `生成递签前内部提醒`

材料检查和建议可以直接返回。生成最终材料包或批量操作需要确认。

## 每日工作简报

每日简报回答一个核心问题：今天团队需要先处理什么。

简报内容：

- 今天递签/面签客户。
- 明天和未来 3 天高风险客户。
- 未来 7 天排期摘要。
- `slotTime` 已有但材料不齐的客户。
- 自动化失败或阻塞的任务。
- 卡点预警：识别停滞案件，例如回执已生成但 24 小时未提交最终表、DS-160 已提交但 3 天未预约。
- 文件导入待确认。
- 缺人、同名、未完成同类案件待选择。
- 长时间未推进案件。
- 常规统计：今日递签数、未来 7 天递签数、待抢号数、资料未齐数、自动化成功/失败数。

排序优先级：

1. 今天/明天递签或面签。
2. `slotTime` 已有但材料不齐。
3. 自动化失败阻塞流程。
4. 案件卡点导致流程停滞。
5. 文件待确认导致案件没建好。
6. 未完成同类案件需要选择归属。
7. 长时间未推进。
8. 常规待办。

简报可以被 Agent 对话查询，也可以在首页或运营工作台默认展示。对卡点预警，简报应直接给出“为什么卡住”的推断和“建议下一步”动作，例如补数据后重试、人工提交、转开发排查或分配给负责人。

队列项类型：

- 缺人文件。

- 同人已有未完成同类案件。
- 文件覆盖确认。

- 自动化启动确认。
- 批量操作确认。
- 材料包生成确认。

每个队列项应显示：

## 风险和缓解

- 风险：文件名解析错误导致建错档。
  - 缓解：导入确认卡、Excel 内容二次校验、低置信度进入待确认队列。
- 风险：同名客户归属错误。
  - 缓解：同名多人必须人工选择；同名单人默认合并但展示确认。
- 风险：覆盖旧材料。
  - 缓解：已有核心文件必须确认替换或另存。
- 风险：Agent 误启动真实自动化。
  - 缓解：所有自动化启动先生成确认卡。
- 风险：提醒误发给客户。
  - 缓解：第一版不做客户发送渠道，内部提醒和客户通知代码路径分离。
- 风险：Codex 被误用为生产运行依赖。
  - 缓解：文档和架构明确 Codex 只用于开发维护，运行时后台任务由网站自己执行。
- 风险：Excel 文件名包含恶意 Prompt 注入指令。
  - 缓解：文件名解析必须在调用大模型前由正则或确定性代码完成，只将解析出的结构化结果放入 Prompt，例如姓名、签证类型、时间窗口和不可用日期；禁止将原始文件名直接作为用户指令拼接给模型。
- 风险：缺乏可观测性导致模型幻觉难以排查。
  - 缓解：第一版必须实现核心调用链日志，记录每次请求的完整 Messages 数组、模型配置、工具候选列表和 Tool 调用轨迹。日志必须脱敏，移除密钥、密码、验证码、护照完整号码和客户敏感联系方式。

## 错误处理

- 文件无法解析：进入缺人或低置信度队列。
- Excel 读取失败：提示文件格式问题，保留原文件，不建档。
- 权限不足：返回明确权限错误，不暴露申请人信息。
- 自动化启动失败：写入任务失败状态，并提供失败摘要。
- 自动化失败摘要必须是人类可理解的失败根因，而不只是技术日志摘要。摘要应说明具体步骤、失败现象、可能原因、专员可执行修正和是否需要开发处理。
- 同步失败：保留旧数据，记录 `lastSyncFailedReason` 或任务错误。
- Agent 工具异常：返回可读错误，允许用户重试或转人工处理。

   - 支持确认、修改后确认、取消。

3. 网站内 AI 对话框。
   - 查询申请人、案件、材料、任务。
   - 修改字段预览和确认。
   - 启动自动化预览和确认。

4. 月历和每日工作简报接入 Agent。
   - 月历高亮内部风险。
   - 今天/未来 3/7 天的工作建议。
   - 未填写递签时间和已完成递签筛选。

5. 自动化结果同步和失败摘要。
   - 法签任务结果写回。
   - 美签任务结果写回。
   - 失败原因转成下一步建议。

暂不实现：

- 客户邮件/微信/短信自动发送。
- 客户侧聊天机器人。
- 外部日历同步。
- Agent 删除申请人或案件。
